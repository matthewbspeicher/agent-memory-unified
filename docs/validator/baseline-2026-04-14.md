## 2. Failure #1 — Evaluator datetime crash (47 crashes / 24h)

**Frequency:** 47 occurrences in the last 24 hours, i.e. once per ~30-minute scheduler tick.

**Trace summary** (full trace in `docker logs`):

```
MinerEvaluator loop error: invalid input for query argument $1:
  '2026-04-14T12:21:02' (expected a datetime.date or
  datetime.datetime instance, got 'str')

File "/app/trading/integrations/bittensor/evaluator.py", line 321, in run
    evaluated = await self.evaluate_pending_windows()
File "/app/trading/integrations/bittensor/evaluator.py", line 50
    windows = await self.store.get_unevaluated_windows(now=now, ...)
File "/app/trading/storage/bittensor.py", line 242, in get_unevaluated_windows
    cursor = await self._db.execute(...)
asyncpg.exceptions.DataError: invalid input for query argument $1
```

**Root cause:** `BittensorStore.get_unevaluated_windows` is passing an ISO-formatted string as the `$1` parameter to asyncpg, which requires a `datetime.datetime` instance (not a string). asyncpg's Postgres codec path does not auto-parse ISO strings the way SQLite text-affinity does.

**Related memory:** `reference_sqlite_datetime_gotcha.md` documents the T-separator vs space-separator SQLite case. This is the asyncpg cousin: on Postgres you must pass `datetime` objects directly, not strings.

**Impact:** Every evaluator tick fails before it can claim any matured window. Even if the scheduler produced forecasts, they would never be scored.

**Proposed fix (needs approval before I touch this):**
- `trading/storage/bittensor.py:242` — convert `now` argument to `datetime.datetime` at the call boundary or, better, ensure the caller passes a `datetime` (evaluator already does — the bug is likely a `str()` somewhere inside the query builder or a stored-string column). 15-30 min investigation before a single-line fix.
- Add a regression unit test in `trading/tests/unit/test_bittensor_store.py`.

## 4. Failure #3 — `WeightSetter` task never started

**Evidence:** `trading/api/app.py:480-516` (the Bittensor lifespan section) registers `bittensor_scheduler` and `bittensor_evaluator` as tasks on `app.state` and `task_mgr`:

```python
app.state.bittensor_scheduler = bittensor_components["scheduler"]
app.state.bittensor_evaluator = bittensor_components["evaluator"]

task_mgr.create_task(
    bittensor_components["scheduler"].run(), name="bittensor_scheduler"
)
task_mgr.create_task(
    bittensor_components["evaluator"].run(), name="bittensor_evaluator"
)
```

**No equivalent block exists for `weight_setter`.** The class is instantiated inside `setup_bittensor` (startup wiring) but the instance is never bound to `app.state.bittensor_weight_setter` nor handed to `task_mgr.create_task`. That is why:

- `/engine/v1/bittensor/metrics` returns a response without a `weight_setter` key (route code at `bittensor.py:212` does `getattr(app.state, "bittensor_weight_setter", None)` and finds `None`).
- Weights are never submitted even when rankings exist.

**Proposed fix:** Add two lines to the lifespan block — `app.state.bittensor_weight_setter = bittensor_components["weight_setter"]` and a `task_mgr.create_task(...)` call. Plus confirm `setup_bittensor` is actually constructing the `WeightSetter` (a quick grep suggests it is, but I have not verified end-to-end).

⚠ This touches `weight_setter.py`-adjacent wiring. Per CLAUDE.md "Ask first" — I am not touching this without your explicit go.
